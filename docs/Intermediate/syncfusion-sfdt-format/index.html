Syncfusion SFDT Format

What is SFDT?​

SFDT (Syncfusion Document Format) is a JSON-based document representation format created by Syncfusion for their Document Editor component.

Why SFDT Exists​

  • Web-friendly: JSON is native to JavaScript
  • Easy manipulation: Modify documents in browser
  • Lightweight: Smaller than full OOXML XML
  • Editor optimized: Built for Syncfusion Document Editor

In AppraiSys​

SFDT is used as an intermediate format:

  1. Templates stored as DOCX
  2. Convert to SFDT for mail merge
  3. Manipulate in Syncfusion
  4. Convert back to DOCX for delivery

Key properties:

  • All measurements in points (72 per inch)
  • Multiple sections supported
  • Headers/footers per section

AppraiSys Usage Patterns​

Converting DOCX to SFDT​

From DocumentToSfdtFormat.cs:

public static string Convert(MemoryStream stream)
{
    // Load DOCX using Syncfusion DocIO
    var doc = new Syncfusion.DocIO.DLS.WordDocument(
        stream, 
        Syncfusion.DocIO.FormatType.Docx
    );
    
    // Convert to SFDT using EJ2 DocumentEditor
    var sfdtDocument = Syncfusion.EJ2.DocumentEditor.WordDocument.Load(doc);
    
    // Optimize output
    sfdtDocument.OptimizeSfdt = true;
    
    // Serialize to JSON
    var sfdt = Newtonsoft.Json.JsonConvert.SerializeObject(sfdtDocument);
    
    // Cleanup
    sfdtDocument.Dispose();
    doc.Dispose();
    
    return sfdt;
}

Converting SFDT to DOCX​

SFDT Limitations and Bugs​

Known Issues (from AppraiSys experience)​

1. Blank Paragraph Artifacts​

Problem: Extra blank paragraphs appear after conversion

Cause: Syncfusion adds spacing paragraphs

Workaround: Clean empty blocks post-conversion

2. Section Break Oddities​

Problem: Section breaks don't convert cleanly

Cause: SFDT section model differs from OOXML

Workaround: Test section-heavy documents carefully

3. Image Embedding Issues​

Problem: External image URLs don't always work

Cause: Syncfusion requires base64 for reliable rendering

Solution: Download and encode images (as in AppraiSys)

4. Header/Footer Preservation​

Problem: Complex headers/footers may lose formatting

Cause: SFDT header model simplified

Workaround: Keep headers/footers simple

5. Content Control Flattening​

Problem: SDTs sometimes removed after conversion

Cause: Syncfusion mail merge behavior

Solution: Set ClearFields = false

6. TOC Refresh​

Problem: Table of contents doesn't update

Cause: SFDT doesn't support automatic TOC update

Workaround: Regenerate TOC in final DOCX

Working with SFDT Programmatically​

Parse SFDT JSON​

using Newtonsoft.Json;
using Newtonsoft.Json.Linq;

var sfdtObject = JObject.Parse(sfdtJson);

// Access sections
var sections = sfdtObject["sections"] as JArray;

// Access first block
var firstBlock = sections[0]["blocks"][0];

// Check if it's a paragraph
if (firstBlock["inlines"] != null)
{
    // It's a paragraph
    var inlines = firstBlock["inlines"] as JArray;
}
else if (firstBlock["table"] != null)
{
    // It's a table
    var table = firstBlock["table"];
}

Modify SFDT​

// Add text to first paragraph
var sections = sfdtObject["sections"] as JArray;
var firstBlock = sections[0]["blocks"][0];
var inlines = firstBlock["inlines"] as JArray;

inlines.Add(new JObject
{
    ["text"] = " Additional text",
    ["characterFormat"] = new JObject()
});

// Serialize back
var modifiedSfdt = sfdtObject.ToString();

Find Text in SFDT​

public List<string> FindAllText(JObject sfdt)
{
    var texts = new List<string>();
    
    foreach (var section in sfdt["sections"])
    {
        foreach (var block in section["blocks"])
        {
            if (block["inlines"] != null)
            {
                foreach (var inline in block["inlines"])
                {
                    if (inline["text"] != null)
                    {
                        texts.Add(inline["text"].ToString());
                    }
                }
            }
        }
    }
    
    return texts;
}

SFDT Best Practices​

✅ DO:​

  1. Optimize SFDT: Always set OptimizeSfdt = true
  2. Validate JSON: Parse to verify valid JSON
  3. Handle images: Convert to base64 for reliability
  4. Test conversions: Always round-trip test DOCX → SFDT → DOCX
  5. Clean empty blocks: Remove artifact paragraphs

❌ DON'T:​

  1. Don't edit SFDT manually: Use Syncfusion APIs
  2. Don't trust external image URLs: Download and encode
  3. Don't ignore validation errors: Fix structure issues
  4. Don't skip round-trip testing: Conversions can lose data
  5. Don't assume OOXML equivalence: Some features differ

SFDT Schema Reference​

Root Level​

{
  "sections": [],           // Required: Document sections
  "characterFormat": {},    // Optional: Default character format
  "paragraphFormat": {},    // Optional: Default paragraph format
  "defaultTabWidth": 36,    // Optional: Default tab width in points
  "trackChanges": false,    // Optional: Track changes enabled
  "styles": [],             // Optional: Style definitions
  "lists": [],              // Optional: List instances
  "abstractLists": [],      // Optional: List templates
  "comments": [],           // Optional: Comments
  "revisions": [],          // Optional: Tracked changes
  "customXml": []           // Optional: Custom XML data
}
